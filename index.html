Graph Databases

  • G = (V, E)
  • Data gets stored on a Vertex
  • Relations are stored as Edges between vertices
  • Edges are direct pointers - no expensive lookup required!

So what is OrientDB?

  • Hybrid Document/Graph Database
  • Schema-less, schema-full, or schema-mixed
  • Inheritance and Polymorphism
  • Extract data using Extended SQL or Gremlin/Blueprints
  • Mult-master replication and sharding
  • ACID transactions
  • APIs for several languages
  • Record level security
  • REST services
  • Embedded functions
  • Browser-based interface
  • Lucene indexes
  • Runs anwyhere there's a JVM
  • ... and more

About the Data

  1. Classes
  2. Document, Vertex, Edge, Property
  3. Clusters

Classes

  • Signifies a collection of data records
  • Analagous to a table in an RDBMS
  • Basic classes in a Graph Database: V, E
  • Allows for inheritance

Document, Vertex, Edge, Property

Document/Vertex

Data entities. Can store structured or unstructured data. Can contain sub-documents.

Edge

Used to store relationships between Vertices. Have in/out directionality.

Property

Typed or JSON data stored in association with Documents, Vertices, or Edges.

Standard Properties

All documents/vertices/edges have some properties

  • @class - String representing the class name
  • @rid - record ID, of the form <cluster>:<id>. This marks the physical location of this node and cannot change.
  • @version - Version count for this element. Stored for transactions and conflict resolution. Old versions are not stored. Mostly won't have to worry about this. Incremented on each update.

Clusters

  • Clusters are associated with classes
  • Every class is created with one cluster (by default)
  • New clusters may be added
  • Each cluster lives on at least one machine (with replication)

Asynchronous Query


 new OSQLAsynchQuery<ODocument>("select from Doc",
                            new OCommandResultListener() {
        @Override
        public boolean result(Object o) {
            ODocument doc = (ODocument) o;
            System.out.println("Received " + doc.getIdentity());
            System.out.println("Name: " +
                doc.field("firstname") +
                " " + doc.field("lastname"));
            return false;
        }

        @Override
        public void end() { }
    }).execute();

Dataflow framework for transforming data through a series

Gremlin

                        
Fluent DSL for graph database traversal

                    


                    

                        
Gremlin-Java

                        

    OrientVertex v = g.getVertex(new ORecordId("15:643"));

    new GremlinPipeline(v).
        in("isMake").
        in("isModel").
        out("Purchased").
        filter(new PipeFunction<Vertex, Boolean>() {
            @Override
            public Boolean compute(Vertex v) {
                BigDecimal price = v.getProperty("price");
                return price.compareTo(new BigDecimal(70_000)) > 0;
            }
        }).in("Bought").property("fullName").
           forEach(System.out::println);

                        


                    

                


                

                    

                        
Security

                    

                    

                        
Schema Security

                        
    
                            
  • Based off of Users and Roles
    • 
                            
  • Users belong to one or more roles
    • 
                            
  • Roles are granted permissions to database-level elements
    • 
                        

                    

                    

                        
Permissions

                        
Similar to Unix rwx permissions, but using CRUD operations

                        
    
                            
  • CREATE - 1 (0001)
    • 
                            
  • READ - 2 (0010)
    • 
                            
  • UPDATE - 4 (0100)
    • 
                            
  • DELETE - 8 (1000)
    • 
                            
  • ALL - 15 (1111)
    • 
                        

                        
May be combined, so READ + UPDATE have a permission value of 6 (0110)

                    

                    

                        
Assigning Permissions

                        
    
                            
  • Roles are granted permissions in two ways:
                                
      
                                    
    • ALLOW ALL EXCEPT
      • 
                                    
    • DENY ALL EXCEPT
      • 
                                
    
                            
    • 
                            
  • Operations are granted based on schema
                                
      
                                    
    • database
      • 
                                    
    • database.class
      • 
                                    
    • database.class.
      • 
                                    
    • database.cluster
      • 
                                    
    • database.cluster.
      • 
                                    
    • database.query
      • 
                                    
    • database.command
      • 
                                    
    • database.config
      • 
                                    
    • database.hook.record
      • 
                                    
    • server.admin
      • 
                                
    
                            
    • 
                        

                    

                    

                        
Example

                        
Enable to the role "motorcyclist" the access to all the classes but the "Car" class:

                        

    update orole put rules = "database.class.*", 15 where name = "motorcyclist"
    update orole put rules = "database.class.Car", 0 where name = "motorcyclist"
                        

                        
Note: resources are case sensitive

Master/Master

                        
    
                            
  • Until 2012, OrientDB used Master/Slave
    • 
                            
  • Fantastic read scalability
    • 
                            
  • All writes went through the master
    • 
                            
  • Reimplemented with Multi-master architecture
    • 
                            
  • Writes can now be processed on all servers
                            
    • Gives almost linear scalability 
    • 
                        

                    

                    

                        
Auto-Discovery

                        
    
                            
  • Configuration is built on top of Hazelcast
    • 
                            
  • Provides auto-discovery using Multicast
    • 
                            
  • EC2 plugin also provided
    • 
                            
  • If multicast is disabled, define IP Addresses
    • 
                            
  • Fantastic for cloud deploys or elastic architectures
    • 
                        

                    

                    

                        
Failover

                        
    
                            
  • Databases can be shared across machines
    • 
                            
  • OrientDB clients are also Hazelcast-aware
    • 
                            
  • If one machine goes down, the rest will continue serving data
    • 
                            
  • Clients connected to that server can automatically reconnect to another server in the cluster
    • 
                        

                    

                    

                        
Sharding

                        
    
                            
  • Data is stored inside clusters
    • 
                            
  • Each class can have multiple clusters
    • 
                            
  • Each cluster can be on a separate physical machine
    • 
                            
  • Clusters may also be replicated to other servers
    • 
                            
  • All done through simple configuration
    •
